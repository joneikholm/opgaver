Adders become a CPU bottleneck

A “correct” adder can still be slow in real hardware — because the carry signal may have to travel through many stages.
Ripple-carry → Critical path

Key idea

In a ripple-carry adder, each bit must wait for the previous carry:

Bit 0 → Bit 1 → Bit 2 → Bit 3 → …
carry → carry → carry → carry → …
Why it matters
  • The slowest route is the carry chain.
  • This sets the CPU’s critical path.
  • Clock speed must allow carry to finish.
Why it shows up everywhere
  • Addition is constant: counters, pointers, addresses.
  • Even “simple” code triggers many adds.
  • Slow add → lots of things slow down.

Scale + solutions

CPUs are often 64-bit.
Naive ripple-carry can chain ~64 stages.
Ripple-carry delay
O(n)
Fast adders (parallel carry)
O(log n)
  • Carry-lookahead: compute carries in parallel.
  • Carry-select / carry-skip: reduce worst-case waiting.
  • Parallel-prefix adders (e.g. Kogge–Stone): very fast, more wiring.
